# v0.15.0 — Team Mode

**Released:** 2026-06-02
**Theme:** A HYBRID dev-team workflow — brainstorm panel, mandatory user decision gate, then parallel dispatch + QA. First Kasidit mode that decides *the approach* before executing it.

## TL;DR

New command `/kasi-team`. Distinct from `/kasi-multi` / `sudo`, which execute a *known* approach — Team Mode exists for missions where the approach itself is the open question.

```
brainstorm panel (default N=3) → 2-3 synthesized options with trade-offs
  → USER DECISION GATE (mandatory)
    → parallel dispatch (reuses /kasi-multi fan-out)
      → mandatory QA pass (audit-specialist --focus=quality)
        → final synthesis
```

## Why this release

`/kasi-multi` and `sudo` are strong when the mission is clear and the only question is execution throughput — fan out N specialists, synthesize. But some missions have real uncertainty about *which approach* to take before any code should be written: architecture choices, migration strategy, "should we build vs buy," or any decision where getting user buy-in before dispatching implementation work saves a costly wrong-direction sprint.

Prior to this release, that judgment either happened informally in the main thread (no structured trade-off comparison, no explicit decision point) or got skipped — the orchestrator would pick an approach and dispatch directly. Team Mode makes the choice-then-execute sequence a first-class, repeatable pattern.

## What's new

### 1. `/kasi-team` command

Full flow:

1. **Brainstorm panel** (default N=3) — each role-agent independently proposes an approach + its top-3 risks. No cross-talk between panelists; independence is the point.
2. **Synthesis** — Main merges the N proposals into 2-3 distinct options with trade-offs (not N options; overlapping proposals get merged, not listed separately).
3. **User decision gate (mandatory)** — options are presented, user picks one. This gate cannot be skipped, including with `--fast` (see below).
4. **Parallel dispatch** — after the pick, Main dispatches implementation specialists in parallel, reusing `/kasi-multi`'s fan-out mechanics (same tier caps, same dispatch brief format from [[v0.14.0]]).
5. **Mandatory QA pass** — `audit-specialist --focus=quality` reviews the dispatched work before final synthesis. Also not skippable with `--fast`.

### 2. CORE + DYNAMIC team composition

- **CORE roles** (always present): Lead = `architect-planner`, QA = `audit-specialist --focus=quality`.
- **DYNAMIC roles** (picked per mission): security/perf/migration lenses for the brainstorm phase, plus whichever implementation specialists the chosen option needs — pulled from the existing Specialist Agent Registry.
- **No new agent files.** Core and dynamic roles are dispatch *personas* over agents that already existed before this release.

### 3. Refinement Counter tuned for the brainstorm phase

The framework-wide Refinement Counter ([[v0.14.0]]) defaults to 3 rounds. Team Mode's brainstorm phase caps it at **1 round** specifically — the goal of brainstorming is convergence to a decision, not endless option refinement. `--fast` skips this one refinement round only; the QA pass is never skipped regardless of flags.

### 4. Haiku tier behavior

- Panel size reduced to N=2 (from default N=3).
- 0 refinement rounds (vs 1 for Sonnet/Opus).
- Dispatch cap N=4 (matches the existing `/kasi-multi` Haiku cap).
- **Refuses** missions that require a security or migration lens — directs the user to Sonnet or Opus tier instead. Team Mode's brainstorm phase involves genuine architectural judgment calls that Haiku's harder rules (no architecture decisions — see [[Model Tiers]]) already preclude it from making solo.

### 5. `sudo` shorthand default bump

Unrelated to Team Mode directly, but shipped in the same release: bare `sudo <mission>` now defaults to **N=6** (matching `/kasi-multi`'s default roster), up from N=2. `sudo <N> <mission>` still lets you override explicitly. `sudo` remains a pacing signal ("skip clarifying questions"), not a permission escalation — this is unchanged.

### 6. Internal hook renaming cleanup

Hook docstrings, comment headers, the `kasi-log.sh` → `kasi-log.py` cross-call reference, `hooks/README.md`, and the `/kasi-init` `settings.json` snippet were updated to consistently use `kasi-*` filenames (the rename from `kasidit-*` shipped functionally in [[v0.11.0]], but some internal references still said the old name). The JSONL emit-token protocol (`[kasidit-log]`, `[kasidit-pattern]`, etc.) is **intentionally unchanged** — those tokens are a stable parser contract for `kasi-record.py`, not a display string.

### 7. Documentation accuracy fixes

- `kasi-review.md` / `kasi-status.md` now show the Sonnet escalation threshold (4 Opus / 3 Sonnet / 2 Haiku) explicitly, instead of omitting the Sonnet row.
- `kasi-devopt.md`'s `infra` row now recommends `audit-specialist --focus=security`; `kasi-multi.md`'s Haiku verifier slot is pinned to `audit-specialist --focus=quality`.
- `kasi-cascade.md` orchestrator paths are now absolute; `kasi-search.md` flags that the embedding layer is optional and not wired into `/kasi-init` — grep fallback is the actually-working path.
- Deprecated-agent stubs (`code-reviewer`, `perf-profiler`, `security-auditor`) reworded — they no longer claim "removed in v0.11." They're kept **indefinitely** for name resolution, which is what was actually true; the doc had been overclaiming a removal that never happened.

### 8. Removed

- Legacy `KASIDIT_CENTER_DIR` fallback dropped from `kasi-drift-check.sh` — it now uses `KASIDIT_CENTER` only, consistent with the other four hooks. The variable was a read-side fallback never seeded by `install.sh`, so no runtime path breaks.
