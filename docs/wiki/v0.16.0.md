# v0.16.0 — Deploy Split

**Released:** 2026-07-09
**Theme:** `/kasi-devopt`'s `deploy <env>` sub-mode splits into two dedicated commands — one that can execute for real, one that permanently never does.

## TL;DR

| Command | Can execute? | Scope |
|---|---|---|
| `/kasi-deploy <env>` | **Yes** — for CLI-native platforms only | Cloudflare (`wrangler`), Vercel, Netlify auto-run behind a confirm gate. Everything else (SSH/VPS, Docker, k8s, Terraform, Serverless, Fly, Platform.sh) is plan-only. |
| `/kasi-review-deploy <env>` | **Never** — any platform, any tier | Read-only preflight checklist. Zero execution, zero file writes. |

`/kasi-devopt`'s old `deploy <env>` sub-mode — which never executed anything, full stop — is superseded by these two. It redirects on invocation; the old flow is kept inline for reference only.

## Why this release

Every prior deploy-adjacent command in Kasidit followed one rule: never execute, always hand the user a plan. That rule was safe but blunt — it treated a `wrangler deploy` (idempotent, versioned, trivially rollback-able via Cloudflare's dashboard) the same as an SSH session against a bare VPS with no rollback tooling at all. As Kasidit users increasingly work with Cloudflare Workers/Pages and Vercel — platforms with a first-party CLI that makes a deploy a single safe, reversible command — the blanket "never execute" rule was leaving real automation on the table for the lowest-risk case while adding no safety for platforms where it mattered.

This release doesn't loosen the safety model — it **partitions** it. Execution is now permitted, but only inside a fixed, auditable capability table, gated harder (typed confirm phrase) exactly where the risk is highest (prod), and with a permanently-safe fallback command (`/kasi-review-deploy`) for anyone who wants zero execution risk, ever, regardless of platform.

## What's new

### 1. `/kasi-deploy <env>` — real execute path

Platform capability table (fixed, not a per-mission judgment call):

| Platform | Detected by | Mode |
|---|---|---|
| Cloudflare Workers/Pages | `wrangler.toml` / `wrangler.jsonc` | **auto-executable** |
| Vercel | `vercel.json` / `.vercel/` | **auto-executable** |
| Netlify | `netlify.toml` | **auto-executable** |
| SSH / bare VPS | no platform file, or `Procfile` | **plan-only** |
| Docker / k8s / Terraform / Serverless / Fly / Platform.sh | `Dockerfile`, `k8s/`, `*.tf`, `serverless.yml`, `fly.toml`, `.platform.app.yaml` | **plan-only** |

Only platforms with a first-party, idempotent, single-command CLI deploy are auto-executable. Everything else gets the exact plan-only behavior `/kasi-devopt deploy` always had.

**Flow:**

1. Confirm env (staging/prod) + service.
2. Run the `/kasi-review-deploy` checklist inline — never skipped.
3. NOT READY verdict → stop. Neither execute nor hand over a plan until preflight passes.
4. Branch on platform capability → auto-execute or plan-only.
5. **Confirm gate:** staging/preview needs a plain confirm; **prod requires the typed phrase `confirm: deploy-prod`** — a plain "yes" is rejected.
6. Dirty working tree → forced to plan-only regardless of env, unless `--allow-dirty`.
7. Execute the single deploy command, capture stdout/stderr/exit code. A nonzero exit is reported, never summarized away.
8. On failure: report the error, stop. **No auto-retry, no auto-rollback** — that's a separate decision for the user (or a follow-up `/kasi-fix` / rollback runbook).
9. Log to `STATE/deploy_history.jsonl` — `mode: executed` with exit code, or `mode: plan-only` with `outcome: handed-to-user`.

### 2. `/kasi-review-deploy <env>` — the never-executes guarantee, extracted

Pulled the preflight checklist out of `/kasi-devopt deploy` into its own command so "zero execution" is a property of the command itself, not a rule embedded in a multi-mode command that also does other things. Checklist:

- Env vars match template?
- Migrations pending?
- Tests passing on target branch? (read CI status if available — never triggers a run)
- Uncommitted changes in the working tree?
- Debug statements left in the diff?
- Rollback plan documented?
- Secrets diff — new/changed keys not in template?

Output: **ready** / **ready with warnings** / **not ready**. Answers "is it safe to deploy", not "how do I deploy" — it never proposes commands to run next, that framing is `/kasi-deploy`'s job.

### 3. Tier gating for deploy execution

Haiku is restricted to the plan-only path on `/kasi-deploy`, on every platform, no exception. Auto-execute requires Sonnet or Opus. `/kasi-review-deploy` has no tier restriction — a mechanical checklist scan carries zero execution risk regardless of which model runs it.

### 4. `/kasi-devopt` deploy sub-mode superseded

The `deploy <env>` row in `/kasi-devopt`'s sub-mode table now redirects to `/kasi-deploy` / `/kasi-review-deploy` on invocation. The original never-executes flow is kept inline in the command file for reference, but is not run for new missions. `/kasi-devopt`'s other sub-modes (`env`, `data`, `infra`, `secrets`, `runbook`, `health`, `connect`) are unchanged.

### 5. Router auto-escalate table updated

`/kasi-deploy` → `full` (prod target additionally forces the typed confirm gate regardless of session mode). `/kasi-review-deploy` → `lite`. `/kasi-deploy` added to the "minimum 2 parallel agents" list alongside the other action-taking heavy commands (`/kasi-review`, `/kasi-security`, `/kasi-fix`, `/kasi-ui`, `/kasi-multi`, `/kasi-cascade`, `/kasi-team`).

## Confirm gate summary

| Env | Auto-exec allowed | Gate |
|---|---|---|
| staging/preview | yes | plain confirm |
| prod | yes, if platform auto-executable | typed `confirm: deploy-prod` + diff shown first |
| any env, dirty tree | no | forced to plan-only, or `--allow-dirty` override |
| any env, preflight NOT READY | no | blocked entirely, fix preflight first |
